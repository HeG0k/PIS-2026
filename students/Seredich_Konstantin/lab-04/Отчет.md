Реализовать **прикладной слой** с разделением операций на **команды** (изменяют состояние) и **запросы** (читают данные) в духе CQRS.

## Ход выполнения работы

### 1. Команды (Commands)

Иммутабельные DTO:

- **`CreateDeckCommand`**
- **`AddCardCommand`**

Валидация примитивов в `__post_init__` (пустые строки и т.д.). Бизнес-инварианты — в домене.

Файлы: [commands.py](src/application/command/commands.py).

---

### 2. Command Handlers

- **`CreateDeckHandler`**, **`AddCardHandler`**.

**Пример потока `AddCardHandler`:** загрузка колоды из репозитория → `deck.add_card()` → `save` → `pull_events()` → публикация через `EventPublisher`.

Команды не «тянут» домен наружу как объект ответа — соответствует CQRS.

---

### 3. Queries и Read DTO

- **`GetDeckByIdQuery`**
- **`DeckReadDto`** — плоская модель для чтения без поведения.

---

### 4. Query Handlers

- **`GetDeckByIdHandler`** — чтение и маппинг домена → DTO, без побочных эффектов.

---

### 5. Application Service (фасад)

**`DeckApplicationService`** — единая точка входа для инфраструктуры: делегирование командам и запросам.

---

### 6. Тестирование

Юнит-тест **`AddCardHandler`** с `unittest.mock`: моки репозитория и `EventPublisher`, проверка порядка вызовов и публикации событий. Файл: [tests/test_add_card_handler.py](tests/test_add_card_handler.py).

## Контрольные вопросы

1. **Command vs Query:** команда меняет состояние; запрос только читает и возвращает DTO.
2. **Почему handler команды не возвращает агрегат:** чтобы не экспортировать доменную модель; данные — отдельным query.
3. **Где валидация:** примитивы в DTO команды; инварианты — в домене.
4. **Query из Command Handler:** избегать «смешивания»; при необходимости — через отдельный read-сервис/проекцию, осознанно.
5. **Request DTO vs Command:** внешний контракт отделён от внутренней команды для эволюции API и безопасности границ.
