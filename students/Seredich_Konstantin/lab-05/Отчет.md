## Ход выполнения работы

### 1. Repository (PostgreSQL)

Реализован **`DeckRepositoryImpl`** поверх интерфейса прикладного слоя:

- **SQLAlchemy ORM** — маппинг доменной сущности на ORM-модель.
- Зависимость от `Session`, передаётся снаружи.

Методы: сохранение и выборка колоды по идентификатору (согласно портам курса).

Файл (адаптер): [infrastructure/adapter/out_bound/deck_repository.py](infrastructure/adapter/out_bound/deck_repository.py).

---

### 2. REST Controller (FastAPI)

**Эндпоинты:**

| Метод | Path | Описание |
|--------|------|----------|
| POST | `/decks/` | Создать колоду |
| GET | `/decks/{id}` | Получить колоду |
| GET | `/health` | Доступность сервиса |

Открытая документация **OpenAPI (Swagger)** по `/docs`.

Файл: [infrastructure/adapter/in_bound/deck_controller.py](infrastructure/adapter/in_bound/deck_controller.py).

---

### 3. БД и Docker Compose

- **PostgreSQL** поднимается через [docker-compose.yml](docker-compose.yml).
- **Alembic** — миграции схемы; первая миграция для таблицы `decks` в [infrastructure/migrations/versions/](infrastructure/migrations/versions/).

---

### 4. Интеграционные тесты

Тест с **testcontainers**: ephemeral PostgreSQL, применение миграций, сохранение и чтение через реализацию репозитория.

Файл: [tests/test_integration.py](tests/test_integration.py).

## Контрольные вопросы

1. **Почему Repository в Infrastructure?** Это технический адаптер к конкретной СУБД; домен зависит только от абстракции порта.

2. **Преимущества ORM:** работа с объектами, меньше ручного SQL, переносимость между СУБД в разумных пределах.

3. **Зачем миграции:** версионирование схемы, воспроизводимые окружения, откаты.

4. **Testcontainers:** реальная СУБД в контейнере повышает достоверность интеграционных тестов по сравнению только с in-memory.
