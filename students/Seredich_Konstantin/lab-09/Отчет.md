## Ход выполнения работы

### 1. Протофайлы (.proto)

Файл: [proto/request_service.proto](proto/request_service.proto)

Определены сообщения `Zone`, `Request`, контракты запрос/ответ для `CreateRequest`, `GetRequest`, `ListRequests`, `ActivateRequest`, а также **server-side streaming** — `StreamActiveRequests` → поток `Request`. Пакет: `pso`, комментарии к полям — в файле.

---

### 2. Генерация кода (protoc / grpcio-tools)

Используется **`python -m grpc_tools.protoc`** (`grpcio-tools`), выходные модули:

- `generated/request_service_pb2.py`
- `generated/request_service_pb2_grpc.py`

Команда и версии зависимостей — в [README.md](README.md) и [requirements.txt](requirements.txt).

---

### 3. gRPC Server

Файл: [grpc/server.py](grpc/server.py)

Реализован класс `RequestServiceServicer`: unary-методы **CreateRequest**, **GetRequest**, **ListRequests**, **ActivateRequest**; **StreamActiveRequests** — цикл `while context.is_active()` с периодической выдачей всех заявок со статусом `ACTIVE` (в учебных целях — «снимок» для мониторинга).

Хранилище — in-memory (как в примере курса).

---

### 4. gRPC Client

Файл: [grpc/client.py](grpc/client.py)

Функции вызова RPC и ограничения потока `stream_active_requests(..., max_messages=...)`. Демонстрация: `python grpc/client.py` при запущенном сервере.

---

### 5. Server-side streaming

Сценарий: клиент вызывает `StreamActiveRequests`, сервер **последовательно отдаёт** сообщения по активным заявкам с паузой между циклами. Проверка — в тестах и при ручном запуске клиента.

---

### 6. Тесты клиента / RPC

Файл: [tests/test_grpc.py](tests/test_grpc.py)

- создание и чтение заявки;
- `NOT_FOUND` для несуществующего id;
- активация `DRAFT` → `ACTIVE`;
- получение **не менее трёх** сообщений из stream активных заявок.

Сервер в фикстуре поднимается на свободном порту: [tests/conftest.py](tests/conftest.py).

## Контрольные вопросы

1. **В чём преимущество gRPC над REST?** Двоичный контракт и эффективный транспорт, строгая схема, удобные паттерны **streaming**, меньше накладных расходов на типичные внутренние вызовы по сравнению с JSON/HTTP.

2. **Почему Protocol Buffers быстрее JSON?** Компактное двоичное представление, без разбора текстовых полей и чисел в runtime так же дорого, как у JSON; быстрее передача по сети и десериализация.

3. **Зачем нужен streaming в gRPC?** Для длительных или частых обновлений (мониторинг, прогресс, события) без polling и без создания множества отдельных HTTP-запросов.

4. **Когда REST, а когда gRPC?** REST чаще для публичных HTTP API, кэшей, браузера; gRPC — для **внутренних** высоконагруженных связей между сервисами и при необходимости потоков.
