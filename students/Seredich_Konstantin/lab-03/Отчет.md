## Цель работы

Применить тактические паттерны DDD (Entities, Value Objects, Aggregates, Domain Events) при реализации **доменного слоя** с инвариантами и доменной логикой.

---

## Вариант №28 — «Говорю красиво» 🗣️

**Питч:** приложение интервального повторения карточек и колод.

**Ядро домена:** студент, карточка, колода; настройки и контент как value objects.

---

## Ход выполнения работы

### 1. Value Objects

Созданы (см. [src/study_service/domain/models/value_objects.py](src/study_service/domain/models/value_objects.py)):

1. **`CardContent`** — термин и определение; валидация в `__post_init__`.
2. **`ReviewGrade`** — оценка ответа (диапазон 1–4).
3. **`DeckSettings`**, **`EmailAddress`** — параметры колоды и контакт.

Иммутабельность: `@dataclass(frozen=True)`.

---

### 2. Entities

Реализованы сущности (см. [entities.py](src/study_service/domain/models/entities.py)):

- **`Student`**, **`Card`**, **`Deck`** — идентичность по `id`, изменение состояния через методы (`add_card`, `archive_deck`), без публичных сеттеров полей.

---

### 3. Aggregate Root

**Корень:** `Deck` инкапсулирует `Card`. Добавление и изменения карточек только через методы колоды.

**Инварианты:**

| №          | Инвариант                         | Проверка |
|------------|-----------------------------------|----------|
| 1          | Нельзя менять архивную колоду    | методы изменения |
| 2          | Непустой контент карточки         | `CardContent` |
| 3          | Нет дубликатов терминов          | `add_card` |
| 4          | Оценка в допустимом диапазоне     | `ReviewGrade` |

---

### 4. Domain Events

Файл: [events.py](src/study_service/domain/models/events.py)

- **`CardAddedToDeckEvent`** — после добавления карточки.
- **`CardReviewedEvent`** — после ответа пользователя (в т.ч. новый интервал).

События накапливаются в агрегате и извлекаются через `pull_events()` для инфраструктурного слоя.

## Контрольные вопросы

1. **VO vs Entity:** у Entity устойчивая идентичность (id); VO определяется всеми атрибутами и обычно неизменяем.
2. **Зачем инкапсуляция агрегата:** единая точка соблюдения инвариантов и граница консистентности.
3. **Роль Domain Events:** зафиксировать факт изменения для других контекстов/проекций без связки домена с инфраструктурой.
4. **Проверка инвариантов:** в конструкторах VO и в методах агрегата перед сменой состояния.
5. **Иммутабельность VO:** предсказуемость, безопасность в качестве ключей/полей, отсутствие скрытых мутаций.
